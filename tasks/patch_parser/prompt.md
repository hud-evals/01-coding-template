# patch-parser

## Overview

**patch-parser** is a Python library for parsing the V4A patch format, a specialized diff notation used by coding agents including codex and cline. The library provides a compact, single-module implementation (455 lines of code) that tokenizes and interprets patch documents conforming to the V4A specification. The parser handles the complete V4A syntax, including patch delimiters (`*** Begin Patch`), file update declarations (`*** Update File: path/to/file.py`), hunk headers with optional context hints (`@@ optional context hint @@`), and line-level operations marked with space (context), minus (removal), and plus (addition) prefixes.

The library is structured around 4 core classes that encapsulate the parsing logic and data representation, complemented by 6 module-level functions that provide the primary API surface. This design enables both low-level access to parsed patch components and high-level convenience functions for common parsing workflows. The implementation is self-contained within a single `patch_parser` module, making it straightforward to integrate into projects that need to consume or process V4A-formatted patches generated by AI coding assistants.

patch-parser is purpose-built for scenarios where coding agents produce structured patch output that must be reliably parsed, validated, and applied to source files. The focused scope and minimal dependencies make it suitable for integration into larger development tools, CI/CD pipelines, and agent-based code generation systems.

## Implementation Notes

### Node 1: Patch Structure and Parsing

The `parse_v4a_patch()` function parses V4A format patches delimited by `*** Begin Patch` and `*** End Patch` markers (both with and without spaces are accepted: `'*** Begin Patch'`, `'***Begin Patch'`, `'*** End Patch'`, `'***End Patch'`). The function returns a tuple of `(List[PatchOperation], Optional[str])` where the error string is `None` on success. Empty input returns an empty operations list with no error. The begin marker is optional; patches can start directly with operation declarations.

### Node 2: Operation Types and File Path Extraction

Four operation types are supported via `OperationType` enum: `ADD`, `UPDATE`, `DELETE`, and `MOVE`. Operations are declared with lines matching patterns:
- `*** Add File: <file_path>`
- `*** Update File: <file_path>`
- `*** Delete File: <file_path>`
- `*** Move File: <old_path> -> <new_path>`

For `MOVE` operations, the `file_path` field contains the source path and `new_path` contains the destination path. For other operations, `new_path` is `None`.

### Node 3: Hunks and Context Hints

Each operation contains a `hunks` list of `Hunk` objects. A hunk is introduced by a context hint line matching `@@ <hint> @@`, which is stored in `context_hint`. If no context hint marker exists, `context_hint` is `None`. Multiple hunks can exist within a single operation, each with its own context hint.

### Node 4: Hunk Lines and Prefixes

Each `Hunk` contains a `lines` list of `HunkLine` objects. Each `HunkLine` has a `prefix` (a single character) and `content` (the line text). Prefixes are:
- `"+"` for added lines
- `"-"` for removed lines
- `" "` (space) for context/unmodified lines

The prefix is extracted as the first character of each line in the hunk; the remainder is the content. Non-prefix characters (e.g., pipes `|`) within the content are preserved as-is.

### Node 5: ADD and DELETE Operations

`ADD` operations contain hunks with lines prefixed by `"+"` representing file content to be created. The `content` field may be populated with full file content. `DELETE` operations typically have no hunks and no content; they only specify the file to remove via `file_path`.

### Node 6: UPDATE Operations

`UPDATE` operations contain hunks with a mix of `" "` (context), `"-"` (removal), and `"+"` (addition) lines. Hunks with only `"+"` lines and a `context_hint` indicate insertion at that location. Hunks with only `"+"` lines and no `context_hint` indicate appending at end of file.

### Node 7: Application Functions

The `apply_v4a_operations()` function applies a list of `PatchOperation` objects using a `file_ops` interface. Individual operation types are delegated to `_apply_add()`, `_apply_delete()`, `_apply_move()`, and `_apply_update()`, each returning a tuple of `(bool, str)` indicating success and a message. The function returns a `PatchResult` object.

### Node 8: Multiple Operations

A single patch can contain multiple operations of different types in sequence. Each operation is parsed independently and added to the operations list in order.
